# 提交代码

## 特性

- 如果没有暂存，则自动暂存变更文件
- 按照不同的关注点及特性拆分提交
- 使用中文，避免使用 emoji 表情，考虑可读性内容精简，突出重点。

## Commit Types

- feat: 新特性
- fix: 问题修复
- docs: 文档或注释变更
- refactor: 没有改变逻辑的代码重构
- style: 例如代码格式化，补充标点等
- perf: 性能优化
- test: 添加或调整测试用例
- chore: 工具、配置变更

## 过程

1. 检查当前缓存的变更
2. 如果没有暂存变更，则对当前的变更文件进行分析
3. 分析变更并确定 commit 类型
4. 如果提交过于复杂，尝试拆分 commit，保证 commit 保持功能的一致性与原子性
5. 生成 commit 的描述
6. 执行提交

## commit 要求

- 忽略合并 master 带来的变更
- 保持 commit 的原子性与专注性
- 以命令式语气编写（例如用'添加功能'而非'已添加功能'）
- 解释为什么，而不只是做了什么
- commit 的内容保持简洁，如果无法一句话描述清楚，说明包含了多条内容，需要拆分 commit
- 将不相关的修改拆分到不同的提交中
- pre-commit 失败不要直接修复，让我选择如何处理
